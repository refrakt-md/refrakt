# v0.27.0 ŌĆö External data & hosted content

Two threads that together let the pipeline ingest **real data from outside the content tree** and
run **without a local filesystem**. The first feeds charts and sortable tables from actual data
files; the second consolidates the remaining ad-hoc fs seams into one injectable interface so a
hosted renderer (the GitHub-app product) can build a tenant's repo straight from the API.

## `data` rune ŌĆö external tabular sources ({% ref "SPEC-103" /%})

A preprocess-time rune that reads an external tabular source (CSV, JSON, NDJSON; SQLite later)
and emits a Markdoc `table` AST node. Because `chart` ({% ref "SPEC-083" /%}) and `datatable`
already treat an authored `<table>` as their single source of truth, a table-emitting `data` rune
feeds **both** ŌĆö plus the bare page ŌĆö with zero changes to either host rune. The external-source
sibling of {% ref "SPEC-062" /%} (`snippet`): one preprocessor resolves a sandboxed file and
replaces the tag with a standard node the container runes consume transparently, and the emitted
table is the honest no-JS fallback for free. {% ref "WORK-417" /%} is already `ready` for this
thread.

## ProjectFiles seam ŌĆö virtual project filesystem ({% ref "SPEC-113" /%})

Consolidates the handful of ad-hoc filesystem seams at the pipeline edges (sandbox example reads,
the snippet/expand/file-ref readers, `fileRoots` scanning, the plan plugin's scan) into one
injectable **`ProjectFiles`** interface with filesystem and in-memory providers. Defines the
fetch-then-build materialization contract for remote sources and closes the sandbox `src`
path-traversal gap as part of the migration. The pipeline is already most of the way there
(`loadContentFromTree` runs the full transform from a caller-assembled tree; core packages are
fs-free) ŌĆö this finishes the job so the hosted product never touches disk.

## Sequencing

The two threads share the **sandboxed-read** surface, which makes {% ref "SPEC-113" /%} the
natural foundation: the `ProjectFiles` interface it defines is the same seam {% ref "SPEC-103" /%}'s
`data` preprocessor reads through. Land the seam first (or at least its read contract), then build
`data` on top of it so the new rune is fs-free and hosted-ready from day one. {% ref "SPEC-103" /%}
can begin against the existing `SandboxHooks` shape if the threads need to overlap.
