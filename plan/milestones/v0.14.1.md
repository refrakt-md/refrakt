# v0.14.1 — Syntax token contract extension

The v0.14.1 milestone implements {% ref "SPEC-056" /%} — widens the `SyntaxTokens` contract from 7 required + 2 optional roles to 7 required + 9 optional, with fallback chains via CSS `var()` so simple presets stay simple and richer presets gain the headroom to faithfully carry well-known palettes (Nord, Dracula, Solarized, Catppuccin, Tokyo Night, GitHub, …) without losing the role splits their authors designed in.

Beyond the syntax foreground, the milestone also formalises two adjacent surfaces: syntax preset modules may *optionally* claim the code surface (`color.code.*`) so canvas-claiming palettes like Nord can ship their canonical bg + fg together; and the tint mechanism gains the ability to project preset modules into scoped CSS classes via `tints[].extends`, enabling live preset showcases in documentation without inventing a new authoring rune.

Nord is imported as the first non-refrakt syntax preset and serves as the validation case for all three extensions — if Nord's 16 hues land on distinct roles where Nord intends them to, if Nord's canonical canvas projects correctly through `color.code.*`, and if Nord renders live on a site whose active preset is niwaki via the scoped-tint mechanism, the spec is right.

A patch milestone scoped to a single spec. Builds on {% ref "SPEC-048" /%} (token contract foundation), {% ref "SPEC-053" /%} (tint shape alignment), and the highlighter-agnostic `--rf-syntax-*` surface SPEC-048 established.

2. **{% ref "WORK-218" /%}** — audit Shiki's stock css-variables theme to confirm which optional roles are already emitted vs. need wiring, then ship the extended css-variables theme in `@refrakt-md/highlight`.
3. **{% ref "WORK-219" /%}** — CSS variable generator emits `var()` fallback chains for each optional role.
4. **{% ref "WORK-223" /%}** — extend `theme.tints[].extends` to accept preset module paths; generator projects scope-eligible namespaces into scoped CSS classes.
5. **{% ref "WORK-220" /%}** — Nord preset module in `packages/lumina/src/presets/nord.ts` as the validation case (extended syntax roles + `color.code.*` canvas).
6. **{% ref "WORK-221" /%}** — Nord preset documentation page with `{% palette %}` blocks tinted via `{% tint preset="nord" %}` and live code blocks rendered through the scoped-tint projection.
7. **{% ref "WORK-222" /%}** — Cross-reference SPEC-056 from SPEC-048 so the foundation spec points to its syntax tier and tint-vocabulary extensions.

WORK-217 and WORK-218 can run in parallel. WORK-219 depends on WORK-217. WORK-223 depends on WORK-219 (it extends the same generator). WORK-220 depends on WORK-217 + WORK-219. WORK-221 depends on WORK-220 + WORK-223 (it's the end-to-end validation page). WORK-222 is independent.

## Out of milestone (deferred)

- Additional imported palettes (Dracula, Solarized, Tokyo Night, Catppuccin, GitHub, One Dark, Monokai) — Nord proves the contract; the rest are mechanical follow-ups in a separate milestone once the shape is validated in practice.
- Per-role `fontStyle` (italic comments, bold keywords) — orthogonal to colour contract, defer to a follow-up spec.
- A preset registry or discovery mechanism — presets remain plain modules imported by path.
- Auto-exposing presets as tints — listing a preset in `theme.presets` does not automatically expose it as a tint; explicit declaration in `theme.tints` is required.
- Extending the tint *body* vocabulary to syntax tokens — tints scope colour identity by deliberate design, but inline syntax authoring belongs to named tints (with `extends`), not the rune body. Locked by SPEC-056's scoping policy.
- Migration of `niwaki` to the extended contract — niwaki is correct at the 9-role tier and stays as-is.
