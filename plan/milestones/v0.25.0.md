# v0.25.0 — Distribution & onboarding

With the theme platform matured across v0.22.0 (type tokens), v0.23.0 (surface axes +
skeleton/skin) and v0.24.0 (live backdrops + motion), the next move is to make that platform
**distributable** and cut the largest activation cost in the toolchain: the blank page. Three
recent specs interlock to take an author from zero to a designed, purpose-built site —
starting templates, robust install from anywhere, and presets shippable independent of a theme.

## Site templates ({% ref "SPEC-109" /%})

Installable, purpose-built site scaffolds — a docs site, an API reference, a changelog, a
portfolio, a landing page — already filled in and wired to the plugins, layouts and theme the
brief needs. Defines the template package format + manifest and the `create-refrakt` changes
that compose a template with a framework and a theme into a working project. The fourth layer
(plugin / template / theme / preset) that makes the whole stack legible.

## Install robustness ({% ref "SPEC-110" /%})

Hardens the shared install surface so a theme **or** template installs cleanly from a
directory, a `.tgz` tarball, or an arbitrary/private registry, into single- **or** multi-site
projects, with consistent name resolution and post-install validation. Closes the stubbed edges
in `theme install` (tarball name resolution, registry selection, `--site` disambiguation) and
adds the parallel templates install path. Packaging infrastructure only — explicitly not
licensing or entitlement. Depends on {% ref "SPEC-109" /%} for the templates shape.

## Decoupled preset packs ({% ref "SPEC-111" /%})

Makes presets a first-class distributable **pack** with explicit **scope**
(foreground/syntax-scoped vs. integrated/palette) and **compatibility** metadata, so any theme
can consume any compatible preset, authors can ship preset collections independent of a theme,
and tooling can list and filter presets by fit. Surfaces the latent mood-vs-skeleton
distinction the engine already enforces. Metadata + packaging + tooling — no engine or
token-merge change.

## Sequencing

1. **{% ref "SPEC-109" /%} (templates)** is the gate — it defines the template package shape
   that {% ref "SPEC-110" /%} then learns to install.
2. **{% ref "SPEC-110" /%} (install)** generalises the existing `theme install` surface to
   cover both themes and templates across all three source shapes.
3. **{% ref "SPEC-111" /%} (preset packs)** runs largely in parallel — it touches the preset
   loader + metadata, not the template/install surface, and converges only at the tooling that
   lists distributable units.

## Before starting

All three specs are still `draft`. Accept each one and break it into work items before
implementation begins — there are no work items scoped to this milestone yet.
