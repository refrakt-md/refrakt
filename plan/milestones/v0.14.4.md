# v0.14.4 — Framework adapter parity with the SvelteKit reference

The v0.14.4 milestone implements {% ref "SPEC-058" /%} — closes the parity gap the v0.14.x patch line opened between `@refrakt-md/sveltekit` (the adapter the refrakt documentation site itself ships on) and the four other framework adapters (`@refrakt-md/astro`, `@refrakt-md/nuxt`, `@refrakt-md/next`, `@refrakt-md/eleventy`) plus the framework-agnostic `@refrakt-md/html` renderer.

Six capabilities currently live only in the SvelteKit plugin and reach every other adapter in this milestone:

1. **Site-level token-overrides CSS** — `composeSiteTokensCss` generates a stylesheet from `theme.presets`, `theme.tokens`, `theme.modes`, and `site.tints` ({% ref "SPEC-048" /%} + {% ref "SPEC-056" /%}). Outside SvelteKit those config fields are silently ignored.
2. **SEO meta enrichment from site config** — `siteName`, `baseUrl`, `defaultImage`, `logo` bake into `og:site_name`, absolute `og:url`, image fallback, and WebSite + Organization JSON-LD. Outside SvelteKit those fields are silently dropped.
3. **CSS tree-shaking by used-rune analysis** — the SvelteKit plugin emits only the per-rune CSS files for runes actually present in the corpus. Outside SvelteKit every site ships the full theme barrel.
4. **Pipeline-stats build summary** — the SvelteKit plugin prints a Phase 1/2/3/4 + warnings table at the end of every build. Other adapters either go silent or inherit the host framework's default output.
5. **`security` (`SecurityPolicy`)** + **`variables` (Markdoc `$name` variables)** plugin options — both are SvelteKit-only surfaces today even though `loadContent` accepts them.
6. **Content HMR** — only SvelteKit watches the content directory + sandbox examples and invalidates the loader cache on `.md` edits.

A patch milestone scoped to application work — no new design, no new contracts. Every capability ships in working form inside `@refrakt-md/sveltekit`; the job is extraction, sharing, and per-adapter wiring along the path each adapter natively uses.

## Implementation order

WORK-239 is the prerequisite for the site-tokens chain; the rest can run in parallel once it lands. WORK-244 (template cleanup) and WORK-246 (pipeline-stats formatter) are independent and shippable any time.

1. **{% ref "WORK-239" /%}** — Extract `composeSiteTokensCss` into `@refrakt-md/transform/node`. Pure relocation; SvelteKit plugin's local copy is removed and re-imported from the shared module. Prerequisite for WORK-240–242 and WORK-245.
2. **{% ref "WORK-240" /%}** — Wire site-tokens CSS through the Astro adapter via a Vite virtual module. Pairs with WORK-241 (the Vite plugin factory can be shared once both wirings exist).
3. **{% ref "WORK-241" /%}** — Wire site-tokens CSS through the Nuxt adapter via a Vite virtual module.
4. **{% ref "WORK-242" /%}** — Surface site-tokens CSS in the three non-Vite adapters (Eleventy passthrough file, Next.js Server Component helper, HTML inline-style helper).
5. **{% ref "WORK-243" /%}** — Thread `SeoToHtmlOptions` (`siteName`, `baseUrl`, `defaultImage`, `logo`) through every adapter's SEO surface.
6. **{% ref "WORK-244" /%}** — Replace `template-astro/src/setup.ts` with a thin `createRefraktLoader` wrapper. Cleanup; independent.
7. **{% ref "WORK-245" /%}** — Bring CSS tree-shaking to every adapter. Builds on WORK-240–242 (the Vite virtual-module helpers gain a sibling `virtual:refrakt/runes.css`; the non-Vite helpers gain a `getUsedCssImports` companion).
8. **{% ref "WORK-246" /%}** — Move the Phase 1/2/3/4 + warnings formatter into `@refrakt-md/content` and adopt it across every adapter's build path. Independent.
9. **{% ref "WORK-247" /%}** — Surface `security` and `variables` options on every non-SvelteKit adapter. Coordinates naturally with WORK-244 (the Astro template's `createRefraktLoader` switch is where the options first land) but doesn't hard-depend.
10. **{% ref "WORK-248" /%}** — Content HMR for Astro + Nuxt (via Vite `configureServer`) and Eleventy (via `addWatchTarget`). Builds on WORK-240, WORK-241, and WORK-244 (the loader's `invalidateSite` callback).
