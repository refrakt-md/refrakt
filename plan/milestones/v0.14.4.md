# v0.14.4 — Framework adapter parity with the SvelteKit reference

The v0.14.4 milestone implements {% ref "SPEC-058" /%} — closes the two parity gaps the v0.14.x patch line opened between `@refrakt-md/sveltekit` (the adapter the refrakt documentation site itself ships on) and the four other framework adapters (`@refrakt-md/astro`, `@refrakt-md/nuxt`, `@refrakt-md/next`, `@refrakt-md/eleventy`) plus the framework-agnostic `@refrakt-md/html` renderer.

Two capabilities currently live only in the SvelteKit plugin:

1. **Site-level token-overrides CSS** — `composeSiteTokensCss` generates a stylesheet from `theme.presets`, `theme.tokens`, `theme.modes`, and `site.tints` ({% ref "SPEC-048" /%} + {% ref "SPEC-056" /%}). Outside SvelteKit those config fields are silently ignored.
2. **SEO meta enrichment from site-level config** — `siteName`, `baseUrl`, `defaultImage`, `logo` bake into `og:site_name`, absolute `og:url` / canonical links, OG image fallback, and WebSite + Organization JSON-LD. Outside SvelteKit those fields are silently dropped.

A patch milestone scoped to application work — no new design, no new contracts. Every capability ships in working form inside `@refrakt-md/sveltekit`; the job is extraction, sharing, and per-adapter wiring.

## Implementation order

WORK-239 is the prerequisite — it ships nothing user-visible but unblocks every wiring item. The five wiring items can run in parallel once WORK-239 lands.

1. **{% ref "WORK-239" /%}** — Extract `composeSiteTokensCss` into `@refrakt-md/transform/node`. Pure relocation; SvelteKit plugin's local copy is removed and re-imported from the shared module. Prerequisite for everything else in the milestone.
2. **{% ref "WORK-240" /%}** — Wire site-tokens CSS through the Astro adapter via a Vite virtual module. Pairs with {% ref "WORK-241" /%} (Nuxt) — the Vite plugin factory can be shared once both wirings exist.
3. **{% ref "WORK-241" /%}** — Wire site-tokens CSS through the Nuxt adapter via a Vite virtual module. Same mechanism as Astro; differs only in registration path (Nuxt module API vs. Astro integration API).
4. **{% ref "WORK-242" /%}** — Surface site-tokens CSS in the three non-Vite adapters (Eleventy passthrough file, Next.js Server Component helper, HTML inline-style helper). Same source, three idiomatic delivery surfaces.
5. **{% ref "WORK-243" /%}** — Thread `SeoToHtmlOptions` (`siteName`, `baseUrl`, `defaultImage`, `logo`) through every adapter's SEO surface so config-driven site_name, absolute URLs, image fallback, and WebSite + Organization JSON-LD work everywhere.
6. **{% ref "WORK-244" /%}** — Replace `template-astro/src/setup.ts` with a thin `createRefraktLoader` wrapper. Cleanup; deletes ~70 lines from the template and fixes its current missing-scoped-tints bug.

## Out of milestone (deferred)

Capabilities the SvelteKit plugin has that this milestone does *not* port to the other adapters, per {% ref "SPEC-058" /%}'s "Out of scope":

- **CSS tree-shaking by used-rune analysis** — {% ref "SPEC-030" /%} explicitly deferred this for non-SvelteKit adapters. The other adapters ship the full theme CSS barrel.
- **Pipeline-stats build summary** — non-Vite adapters have no equivalent natural hook for the Phase 1/2/3/4 + warnings table the SvelteKit plugin prints.
- **`security` option (SecurityPolicy)** — only consumed by `loadContent` in the SvelteKit virtual content module today. Adapt later as a follow-up.
- **`variables` (Markdoc `$name` variables)** — same shape; adopt later if a non-SvelteKit consumer asks.
- **Content HMR for non-Vite frameworks** — {% ref "SPEC-030" /%} deferral.
