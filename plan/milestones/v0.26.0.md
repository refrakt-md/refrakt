# v0.26.0 — Editorial reading & shared layout modes

Continues the {% ref "SPEC-094" /%} theme-platform arc by settling the two capability gaps it
deferred: the reading register for body text (the last reserved §8 candidate) and the promotion
of `carousel` from a `gallery`-coupled behavior to a canonical layout mode any rune can opt into.
Where v0.23.0 governed a rune's **container** (elevation/prominence/width), this milestone reaches
**inside** it — how the running text reads, and how homogeneous items lay out.

## Reading role ({% ref "SPEC-108" /%})

A `reading` role on body content, emitted as a refinement of `data-section="body"`
(`data-reading="<value>"`) rather than a new top-level axis — settling the exact seam
{% ref "SPEC-094" /%} §8 named. A small semantic register vocabulary (tightest-UI →
most-editorial) drives the editorial-text cluster: measure, paragraph rhythm, drop-cap
eligibility, the styled opening. The author/layout/rune default picks the register; the theme
owns the magnitude. This is arguably the single largest differentiator between a product/docs
theme and an editorial one, and today there's no attribute to hang it on.

## Carousel as a shared layout mode ({% ref "SPEC-100" /%})

Promotes `carousel` to a canonical layout mode under {% ref "ADR-018" /%} — the first value to
graduate into the shared pool (3+ consumers: `feature`, `testimonial`, `pricing`, `cast`, beyond
today's `gallery`). Generalises `gallery`'s carousel into a block-agnostic, contract-bound
behavior, defines the shared track/item DOM contract behind the `carousel` token, then adopts it
across candidate runes starting with `feature`. Same shape as `tabs`/`accordion`/`datatable`:
identical content, a progressive-enhancement behavior layer.

This thread has a **hard prerequisite**: {% ref "SPEC-099" /%} (`feature`'s `layout` axis +
responsive `collapse`) must land first — SPEC-100 adopts carousel onto `feature` and builds its
collapse-to-carousel target on the `collapse` breakpoint SPEC-099 introduces. That work is not
done (`feature` has no `layout` attribute today), so SPEC-099 is pulled into this milestone ahead
of SPEC-100's Phase B.

## Sequencing

The two epics are independent and can run in parallel — one is a theme/CSS-register concern,
the other a behavior/layout-contract concern. {% ref "SPEC-108" /%} is the higher-value of the
two (it closes the theme-platform epic); the carousel epic is a clean, well-scoped generalisation
with an existing reference implementation in `gallery`.

Within the carousel epic the order is fixed: {% ref "SPEC-099" /%} (feature `layout` + `collapse`)
→ {% ref "SPEC-100" /%} (generalise the behavior + contract, then adopt). Phase A of SPEC-100 (the
attribute-dispatch path, the shared contract, lifting the behavior out of `gallery`) does **not**
depend on SPEC-099 and can proceed alongside it; only Phase B (adoption onto `feature`) needs
SPEC-099 landed.

## Before starting

All three specs ({% ref "SPEC-099" /%}, {% ref "SPEC-100" /%}, {% ref "SPEC-108" /%}) are `draft`.
Accept each and break it into work items before implementation — no work items are scoped to this
milestone yet.
