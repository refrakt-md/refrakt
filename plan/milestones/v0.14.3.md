# v0.14.3 — Navigation enrichment: slug resolution and mega menus

The v0.14.3 milestone delivers two related navigation specs: {% ref "SPEC-055" /%} fixes nav slug ambiguity and the multiple-items-active-at-once symptom by moving slug resolution to build time and introducing multi-segment slugs as the first-class disambiguator; {% ref "SPEC-054" /%} adds a `layout="mega"` value to the nav rune for richly-presented header dropdowns (multi-column groups, per-item descriptions, featured hero entries, footer band) and ships a new core `{% badge %}` inline rune as a side benefit.

SPEC-055 is foundational — it has to land first because the mega variant's enrichment is much cleaner once URL resolution is build-time and unambiguous. SPEC-054 layers on top, reusing the slug resolution path and the existing `auto=true` frontmatter enrichment machinery.

A patch milestone scoped to navigation polish. Builds on {% ref "SPEC-046" /%} (which defined the original `layout` attribute and deferred mega menus + per-item enrichment to future work).

## Implementation order

SPEC-055 work items land first as the foundation. SPEC-054 items can begin once description resolution generalises, but the mega layout work depends on SPEC-055 being merged.

1. **{% ref "WORK-231" /%}** — Build-time slug resolution + error system (the foundational refactor; everything else assumes resolved hrefs in SSR HTML)
2. **{% ref "WORK-232" /%}** — Build-time active state + Lumina CSS (depends on WORK-231 for resolved hrefs)
3. **{% ref "WORK-233" /%}** — Site nav migration + authoring docs (verifies existing navs still work, documents the new rules)
4. **{% ref "WORK-234" /%}** — `{% badge %}` core inline rune (independent — can ship in parallel with anything above)
5. **{% ref "WORK-235" /%}** — Description / icon enrichment generalised across `auto=true` layouts
6. **{% ref "WORK-236" /%}** — Mega layout: engine + Lumina CSS + behavior extension (depends on WORK-234 + WORK-235)
7. **{% ref "WORK-237" /%}** — Mega authoring docs + site demo (lands last as the editorial pass)

## Out of milestone (deferred)

- **`base="/path"` attribute on `{% nav %}`** — SPEC-055 explicitly defers this until a use case appears. Nav-location-relative resolution plus multi-segment slugs cover the realistic patterns; adding the attribute is YAGNI today.
- **`nav-key` frontmatter** — SPEC-055 rejects a second identifier system. If renaming-heavy or very deep URL paths force the issue later, revisit in a separate spec.
- **Embedded media in mega panels** — images, video thumbnails as featured entries. SPEC-054 reserves the featured slot for blockquote-as-link in v1; richer media can land in a later spec.
- **Nested groups in mega panels** — sub-groups inside a column. Single-level groups with `---` column splits cover realistic patterns; deeper nesting is YAGNI.
- **Aggregated/auto-populated footer band** — e.g. last 3 changelog entries auto-pulled. Footer band is hand-authored in v1; a future `{% nav-recent %}` primitive could populate it.
- **Per-trigger panel routing logic** — explicit `for="trigger-slug"` attributes on groups. v1 uses slug match (`- product` matches `## Product`); explicit routing is the escape hatch for collisions if they appear.
- **Migration tool for ambiguous slugs** — SPEC-055 deliberately ships the build error as the migration tool. A custom CLI command is unnecessary ceremony.
- **Custom active-state matching rules** — exact + longest-prefix covers every realistic case. Regex / glob matching is over-engineering for a nav primitive.
- **A new declarative content-model primitive (`delimitedSections`)** — investigated in conversation; deferred. Nav's `headingsToList()` imperative path is already in place and well-understood. Adding a framework primitive for one consumer is premature; revisit if a second rune needs the same shape.
